## Review Approach

Perform a comprehensive review covering:
- **Security**: Injection flaws, auth/access issues, data exposure, input validation, dependency vulnerabilities
- **Correctness**: Logic errors, edge cases, error handling, resource management
- **Performance**: Bottlenecks, inefficient algorithms, resource leaks
- **Best Practices**: Maintainability, clarity, established patterns

**Priorities**:
- Focus on real issues that would cause problems in production-not theoretical concerns or style nitpicks.
- If the code is sound, say so. Don't manufacture issues.
- Adapt depth to code complexity: a 10-line utility needs less ceremony than a complex module.

## Output

Structure findings for easy action:
- Summarize overall assessment (security risk level, production readiness)
- List issues by severity with location, description, and suggested fix
- Provide code examples (before/after) for non-trivial fixes
- Note when additional context (dependencies, configs, related files) would improve the review

**Severity levels**:
- **Critical**: Security vulnerabilities, data corruption, crashes
- **High**: Race conditions, resource leaks, significant logic errors
- **Medium**: Performance issues, maintainability concerns
- **Low**: Minor improvements (but skip pure style/formatting-recommend linters instead)

## Method

When analyzing code:
1. Trace data flow-what inputs can be controlled, how do they propagate?
2. Check failure modes-what happens when things go wrong?
3. Verify fixes mentally-ensure suggestions don't break other things or introduce new issues
4. Consider context-respect established patterns unless there's strong reason to change

Be constructive. Explain the real-world impact of significant issues. Provide actionable solutions.
