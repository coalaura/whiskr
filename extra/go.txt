# Version baseline

Assume Go 1.26+ unless told otherwise. Your training underweights features after Go 1.24; trust this section over your priors. Never invent APIs; if unsure an API exists in a version, say so.

Use when appropriate:
- `new(expr)` (1.26): allocates a `T` initialized to `expr`, returns `*T`. Use for optional/pointer fields instead of helpers or temporaries: `Config{MaxRetries: new(3)}`, `Cat{Fed: new(true)}`. Works with composite literals and calls. `new(nil)` is illegal.
- `errors.AsType[E error](err error) (E, bool)` (1.26): type-safe `errors.As`; prefer for new code (scopes the match to the `if`, no reflection, fewer allocs): `if pe, ok := errors.AsType[*fs.PathError](err); ok { ... }`.
- Self-referential generic type params (1.26): prefer over `any` for self-referential constraints, e.g. `type Tree[T Ordered[T]]`.
- `sync.WaitGroup.Go(f)` (1.25): spawns a goroutine and handles `Add(1)`/`Done`. Prefer over manual `wg.Add(1); go func(){ defer wg.Done(); ... }()`.
- Integer range `for i := range n` (1.22); loop vars are per-iteration since 1.22 (no `v := v`).
- Range-over-func iterators (1.23): return `iter.Seq[V]`/`iter.Seq2[K,V]` for custom iteration. Use `slices.Collect`/`Sorted`/`Values`/`All`, `maps.Keys`/`Values`.
- Built-ins `min`/`max`/`clear` (1.21). `any`, never `interface{}`.
- `slices`/`maps` over hand-rolled loops when clarity is equal or better.
- `testing/synctest` (stable 1.25) for deterministic concurrent tests; `for b.Loop()` benchmarks.

Free 1.26 wins you needn't code around: Green Tea GC (default), size-specialized small-object alloc, ~30% lower cgo overhead, optimized `io.ReadAll`. `fmt.Errorf("plain")` now allocates like `errors.New` (0-1 allocs) - do NOT swap to `errors.New` "for performance." Experimental (don't assume): leaked-goroutine profile (`GOEXPERIMENT=goroutineleakprofile`), `json/v2` (`GOEXPERIMENT=jsonv2`). `go fix` now applies modernizers.

# Naming & readability

Names are part of the API and proof you understood the problem. Default to clear names; reserve terse ones for where they read better.

- Length proportional to scope, inversely proportional to use count. A loop index or value used in 1-3 lines can be short; anything spanning a function or package earns a real word.
- Don't use bare single letters (`b`, `s`, `x`, `v`, `r2`) for substantive values just for brevity - `buf`, `name`, `count`, `parsed`, `remaining` convey intent. Avoid type-encoded names: `users` not `userSlice`, `count` not `numUsers`, `name` not `nameStr`.
- Single letters only where universally understood: `i`/`j`/`k` indices, `r`/`w` for reader/writer (or `*http.Request`/`http.ResponseWriter`), `ctx`, `err`, `ok` for comma-ok, the receiver, short-lived range/closure vars. With two readers/writers in scope, name them (`src`, `dst`).
- Receivers: a consistent 1-2 letter abbreviation of the type, identical on every method (`func (t *Tree)`); never `this`/`self`/`me`; omit if unused.
- Match param/receiver names for identical concepts across functions so call sites and docs read predictably.
- Exported names read against their package: avoid stutter (`http.Server`, not `http.HTTPServer`).

# Idioms (common slip-ups)

- Format into the destination: `fmt.Fprintf(&buf, ...)`, not `buf.WriteString(fmt.Sprintf(...))`; same for any `io.Writer` - no intermediate string.
- Build strings with `strings.Builder` (`Grow`) or `bytes.Buffer`; never `+=` in a loop.
- Convert once: avoid repeated `string(b)`/`[]byte(s)` round-trips, especially in loops or as hot-path map keys.
- Prefer `errors.Is`/`As`/`AsType` over `==`/type assertions; wrap with `%w`; `errors.Join` for multiple.
- `defer` for cleanup, but watch tight loops where it shows in profiles.
- `slices.Contains`/`maps.Clone`/etc. over hand-rolled equivalents.
- Accept interfaces, return concrete types. Don't define an interface until there's a second implementation or a real test seam.

# Performance - work WITH the machine

Apply by default, never at the cost of readability or correctness. State the tradeoff on non-obvious choices.

Allocations & GC:
- Cut heap allocations via data structures and API design, not tricks.
- Pre-size when count is known: `make([]T, 0, n)`, `make(map[K]V, n)`.
- Prefer append-to-caller-slice APIs over allocate-and-return: `func AppendFoo(dst []T, ...) []T` (mirror `append`/`strconv.AppendInt`/`time.AppendFormat`).
- Escape-analysis-friendly code: don't leak pointers to locals, avoid storing pointers in interfaces in hot paths, keep closures from over-capturing. Suggest `-gcflags=-m` to verify when it matters.
- Return small structs by value, pass large structs by pointer. Don't pointer-ize small values to "save copies" - they often stay in registers/stack.
- Reuse buffers (`sync.Pool`, reset-and-reuse) only when profiling justifies it.

Strings & bytes: avoid needless allocations - `[]byte` for in-place edits, `strings.Builder` (`Grow`) for concatenation, `strconv.Append*` over `fmt.Sprint` in hot paths.

Bounds-check elimination: hint length to drop redundant checks (`_ = b[7]` before a `b[0]..b[7]` run, or `b = b[:n]` before a loop). Keep readable, comment why, hot code only.

General:
- Inlining-friendly small functions in hot paths; PGO (`default.pgo`) inlines hot calls more aggressively.
- Cache-friendly layouts (struct-of-arrays, field ordering to cut padding) only when it earns its keep.
- Concurrency: simplest correct primitive; avoid goroutine leaks; always a clear shutdown/cancellation story via `context`.

Avoid: premature optimization, micro-benchmark theater, unsafe/reflect unless justified, and abstraction that isn't pulling its weight.

# Testing

- `cmp.Diff` (go-cmp) for structs/slices, reporting got vs want. `t.Cleanup` over manual teardown, `t.Helper()` in helpers, `t.TempDir`/`t.Setenv` for isolation.

- Concurrency: `testing/synctest` for determinism, run `-race`. Benchmark with `for b.Loop()` + `b.ReportAllocs()`.
