## Engineering standards
- Write the best-quality code possible: correct, readable, and maintainable, with sound error handling and clear control flow (return early instead of deep nesting).
- For expected failures (I/O, parsing, validation) return `nil` plus an error message; use `error()`/`assert()` only for programmer errors and contract violations.
- Always declare `local`; never leak globals. Keep variable scope as small as possible. Don't use `<const>` and `<close>`.
- Favor pure Lua over expensive abstractions in hot paths; cache repeated lookups. Lift constant tables out of loops to avoid needless allocations. Don't micro-optimize at the cost of clarity.
- Do NOT cache native or stdlib methods into locals (e.g. `local sin = math.sin` or `local GetGamePool = GetGamePool`). The performance gain is negligible and it only makes the code uglier and harder to read.
- Do NOT wrap `:` (colon) method calls in standalone helper functions. This adds zero benefit and just produces uglier indirection; call the method directly.

## Naming & style
- `camelCase` for variables and functions, `TitleCase` for constants. Names must be descriptive and readable; never use 1-2 letter names (loop counters and idiomatic short names like `k, v`/`i` are fine).
- Prefer clean, conventional idioms over uglier "faster" ones: use `table.insert`/`table.remove` rather than `t[#t+1]` and `t[i]=nil` micro-optimizations, since the gain is negligible and the cost in readability is real.
- Add comments ONLY where code does something non-obvious that a future maintainer would otherwise misread. Otherwise write self-documenting code with no comments.
- If the user provides code, match their existing style, formatting, and conventions over your own defaults.

## FiveM / CfxLua
When working in a FiveM context:
- Use the `Citizen.` namespace (`Citizen.CreateThread`, `Citizen.Wait`, `Citizen.SetTimeout`) rather than the bare aliases.
- Never call `Citizen.Wait` with a value between 1 and 50. Below ~50ms timing becomes unreliable since a client's FPS may drop under that threshold, producing inconsistent wait durations. When you need the tightest loop, use `Citizen.Wait(0)` to cleanly wait exactly one tick instead; otherwise use 50ms or higher with adaptive backoff when idle.
- Use `PlayerPedId()` instead of `GetPlayerPed(-1)`.
- Use `#(vecA - vecB)` for distance instead of `Vdist`/`GetDistanceBetweenCoords`; leverage native vector3/quaternion types.
- Use `GetHashKey()` instead of compile-time backtick hashes (`` `model` ``) or `joaat()`.
- Some natives are costly: cache their computed *results* (e.g. an entity handle or a position) outside hot loops and avoid redundant calls per frame. This means reusing returned values, NOT aliasing the native function itself into a local.
- Treat the client as untrusted. Validate and enforce all authoritative logic (money, items, permissions, positions) server-side; never trust client-sent values.
